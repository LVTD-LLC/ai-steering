# AGENTS.md

This is the vendor-neutral operating manual for AI agents working in this repository. Read it before editing code or docs, then use the more specific steering files when they apply.

## Project

AI Steering Files is a static Astro site that catalogs useful context and steering files for AI coding agents.

## Steering File Map

- `AGENTS.md`: repo-wide instructions for all coding agents.
- `PRODUCT.md`: product purpose, audience, scope, and success criteria.
- `TECH.md`: stack, commands, architecture, and deployment constraints.
- `STRUCTURE.md`: where files live and where new files belong.
- `VISION.md`: durable direction and non-goals.
- `DESIGN.md`: visual and UX rules for this technical index.
- `CLAUDE.md`, `GEMINI.md`, `.github/copilot-instructions.md`, and `.cursor/rules/*.mdc`: tool-specific entry points. Keep shared rules in the agent-agnostic files above.

## Source of Truth

- `src/data/steering-data.js` is the canonical catalog data used by the site and generated README.
- `src/data/resources.ts` validates and exports that data for Astro.
- `README.md` is generated by `scripts/generate-readme.mjs`; when catalog entries change, edit `src/data/steering-data.js` and run `npm run sync:readme`.
- Do not manually duplicate resource guidance in multiple places unless it is intentionally tool-specific.

## Commands

- Install dependencies: `npm install`
- CI-style install: `npm ci`
- Run local dev server: `npm run dev`
- Check and build: `npm run build`
- Astro-only check: `npm run check`
- Regenerate README from catalog data: `npm run sync:readme`

## Workflow

- Prefer small, focused edits.
- Preserve exact file names and capitalization for steering files.
- Use `rg` or `rg --files` when searching the repo.
- After changing catalog data, run `npm run sync:readme` before validation.
- Before finishing a code or content change, run `npm run build` unless there is a clear environment blocker.
- Do not commit generated `dist/`, `.astro/`, `node_modules/`, or local environment files.

## Content Rules

- Keep advice practical and repo-oriented.
- Prefer primary references and concrete examples over generic AI commentary.
- Explain what each steering file is for, what it should contain, and when it is worth adding.
- Avoid duplicating long reference text from third-party docs; summarize and link.
- Keep file names and paths exact, including capitalization.
- Treat external examples as curated recommendations; remove links that are broken, speculative, or no longer representative.

## Design Rules

- The site should feel like a useful technical index, not a SaaS landing page.
- Use restrained colors, readable cards, and dense but scannable content.
- Avoid decorative gradients and oversized marketing sections.
- Keep the first screen focused on browsing the catalog, not marketing copy.
- Make mobile layouts readable without horizontal scrolling or overlapping text.

## Deployment

The site builds to static HTML in `dist/` and includes Docker, nginx, and CapRover config for deployment to `ai-steering.lvtd.dev`.

## Risk Boundaries

- Do not change deployment files (`Dockerfile`, `nginx.conf`, `captain-definition`, `.github/workflows/ci.yml`, or `.github/workflows/deploy.yml`) unless the deployment behavior is part of the task.
- Do not add a backend, database, analytics, or external runtime dependency without a clear product reason.
- Do not replace the generated README workflow with hand-maintained duplicate content.
